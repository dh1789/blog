# PRD: 자동 번역 시스템

**문서 번호**: 9.0
**작성일**: 2025-11-04
**대상 독자**: 주니어 개발자
**우선순위**: High
**예상 공수**: 5-6일

---

## 1. 소개/개요

### 배경

현재 블로그는 한국어(ko)와 영어(en) 두 가지 언어로 콘텐츠를 제공합니다. Polylang 플러그인을 통해 다국어 지원이 가능하지만, 수동으로 영어 번역을 작성하고 발행하는 과정이 시간 소모적이며 일관성 있는 품질 유지가 어렵습니다.

**현재 워크플로우 문제점**:
1. 한글 포스트 작성 → 발행
2. 별도로 영어 번역 작성 (수동)
3. 영어 포스트 발행 (수동)
4. Polylang 언어 연결 (수동)
5. 번역 품질 일관성 유지 어려움
6. 시간 소모: 포스트 1개당 2-4시간 추가 소요

### 해결책

한글 포스트 발행 시 자동으로 고품질 영어 번역을 생성하고 발행하는 시스템을 구축하여 콘텐츠 제작 시간을 80% 단축하고 번역 품질 일관성을 확보합니다. Claude Code를 활용하여 자연스러운 영어 번역을 제공하며, SEO 최적화된 콘텐츠를 자동으로 생성합니다.

### 목표

- 콘텐츠 제작 시간 80% 단축
- 번역 품질 일관성 확보 (Claude API 활용)
- 다국어 콘텐츠 발행 프로세스 완전 자동화
- SEO 최적화된 번역 제공

## 2. 목표

1. **생산성 향상**: 포스트 1개 발행 시간을 4시간에서 30분으로 단축 (85% 개선)
2. **품질 보증**: 번역 품질 검증 통과율 ≥95% 달성
3. **완전 자동화**: 수동 개입 없이 한글→영어 전체 워크플로우 완료
4. **일관성 유지**: 전문 기술 용어 번역 일관성 100% 달성
5. **SEO 최적화**: 영어 SEO 키워드 보존율 100% 달성

## 3. 사용자 스토리

### 주요 사용자 스토리

**As a** 블로그 운영자
**I want to** 한글 포스트 발행 시 자동으로 영어 번역이 생성되고 발행되기를
**So that** 다국어 콘텐츠 제작 시간을 단축하고 일관된 품질을 유지할 수 있다

**수용 기준**:
- `blog publish` 명령어로 한글 포스트 발행 시 영어 번역 자동 생성
- 번역된 영어 포스트가 자동으로 WordPress에 발행됨
- Polylang 언어 연결이 자동으로 설정됨
- 번역 품질 검증 통과 (라인 수 ±50%, SEO 키워드 유지, 제목 길이 <60자)
- 번역 실패 시 한글 포스트만 발행되고 명확한 에러 메시지 출력

### 보조 사용자 스토리

#### 스토리 1: 번역 품질 검증

**As a** 블로그 운영자
**I want to** 번역 품질이 자동으로 검증되기를
**So that** 저품질 번역이 발행되는 것을 방지할 수 있다

**수용 기준**:
- 번역 후 라인 수가 원본의 50-150% 범위 내
- 주요 SEO 키워드가 번역 후에도 유지됨
- 영어 제목 길이가 60자 이하
- 기본 문법 및 오타 검증

#### 스토리 2: 실패 처리

**As a** 블로그 운영자
**I want to** 번역 실패 시에도 한글 포스트는 정상 발행되기를
**So that** 번역 문제로 인해 전체 발행이 중단되지 않도록 한다

**수용 기준**:
- 번역 API 호출 실패 시 한글 포스트만 발행
- 번역 품질 검증 실패 시 한글 포스트만 발행
- 실패 원인이 명확하게 로그에 기록됨
- 사용자에게 명확한 에러 메시지 제공

#### 스토리 3: 번역 범위 제어

**As a** 블로그 운영자
**I want to** 제목, 본문, 요약, 카테고리, 태그가 모두 번역되기를
**So that** 완전한 다국어 콘텐츠를 제공할 수 있다

**수용 기준**:
- 제목(title) 번역
- 본문(content) 번역
- 요약(excerpt) 번역
- 카테고리(categories) 번역
- 태그(tags) 번역
- frontmatter의 모든 번역 가능한 필드 처리

## 4. 기능 요구사항

### FR-1: 번역 트리거

**FR-1.1**: CLI `blog publish` 명령어에 자동 번역 기능 통합
- 한글 포스트(`language: ko`) 발행 시 자동으로 영어 번역 생성
- 기본적으로 자동 번역 활성화
- `--no-translate` 플래그로 자동 번역 비활성화 가능

**FR-1.2**: 번역 필요 여부 자동 감지
- frontmatter의 `language` 필드 확인
- `language: ko`인 경우에만 번역 수행
- 영어 포스트(`language: en`)는 번역하지 않음

### FR-2: 번역 엔진

**FR-2.1**: Claude Code 통합 (이미 구현된 기능 활용)
- **기존 구현 활용**: `packages/core/src/claude.ts`의 `executeClaude()` 함수 사용
- **headless 모드**: Claude Code CLI를 child_process로 실행
- **API 키 불필요**: 로컬에 연동된 Claude Code 구독 자동 활용
- **모델**: Claude 3.5 Sonnet (Claude Code 기본 모델)

**FR-2.2**: 번역 프롬프트 최적화

번역 품질의 핵심은 프롬프트 설계입니다. 다음 요소들을 포함한 상세한 프롬프트를 작성해야 합니다:

**프롬프트 구성 요소**:

1. **역할 정의**:
   - "You are a professional technical writer and translator specializing in developer documentation"
   - 기술 블로그 맥락 이해 필수

2. **번역 목표**:
   - 한국어 기술 블로그 → 영어권 개발자 대상 자연스러운 영어로 변환
   - 직역이 아닌 의역 (문화적 맥락 고려)

3. **품질 기준**:
   - **자연스러움**: 원어민이 읽기 자연스러운 표현 사용
   - **전문성**: 기술 용어의 정확성 및 일관성 유지
   - **가독성**: 문장 구조 최적화 (긴 한국어 문장 → 짧은 영어 문장으로 분리)
   - **톤**: 전문적이면서도 친근한 블로그 톤 유지

4. **필수 보존 항목**:
   - SEO 키워드 (frontmatter `tags` 필드)
   - 코드 블록 (```) 내용 및 구조
   - 링크 URL 및 형식
   - 이미지 경로
   - 마크다운 서식 (헤딩, 리스트, 테이블 등)

5. **번역 금지 항목**:
   - 브랜드명: "Cloudflare", "WordPress", "Namecheap" 등
   - 기술 용어: "CDN", "SSL", "DNS", "API" 등
   - 명령어: `npm install`, `git commit` 등
   - 파일명/경로: `package.json`, `/src/components/` 등

6. **문화적 맥락 변환**:
   - 한국어 관용구 → 영어권 독자가 이해할 수 있는 표현으로 대체
   - 한국 특화 예시 → 글로벌 예시로 일반화
   - 존댓말 → 전문적이지만 친근한 톤으로 변환

**FR-2.3**: 번역 범위
- `title`: 포스트 제목
- `content`: 마크다운 본문
- `excerpt`: 포스트 요약
- `categories`: 카테고리 목록
- `tags`: 태그 목록

### FR-3: 번역 품질 검증

**FR-3.1**: 라인 수 검증
- 번역 후 라인 수가 원본의 50-150% 범위 내
- 범위 초과 시 경고 및 실패 처리

**FR-3.2**: SEO 키워드 검증
- 원본에 포함된 주요 키워드가 번역본에도 존재
- 키워드 추출: frontmatter의 `tags` 필드 활용
- 누락된 키워드가 있을 경우 경고

**FR-3.3**: 제목 길이 검증
- 영어 제목 길이 ≤60자 (SEO 최적화)
- 초과 시 경고 (발행은 진행)

**FR-3.4**: 기본 검증
- 번역 결과가 비어있지 않음
- 코드 블록이 손상되지 않음 (``` 개수 일치)
- 링크가 깨지지 않음 (URL 형식 유지)
- 마크다운 헤딩 구조 보존 (h2, h3 등)

**FR-3.5**: 자연스러움 검증 (선택적)

자동 검증이 어려운 항목으로, 향후 LLM 기반 품질 평가 도입 고려:

- **문장 자연스러움**: 기계 번역 특유의 어색한 표현 감지
- **기술 용어 적절성**: 전문 용어의 올바른 사용
- **톤 일관성**: 블로그 특유의 친근하면서 전문적인 톤 유지

### FR-4: 자동 슬러그 처리

**FR-4.1**: 영어 슬러그 자동 생성
- 원본 한글 슬러그에 `-en` 접미사 자동 추가
- 예: `domain-cloudflare-ssl-setup` → `domain-cloudflare-ssl-setup-en`
- 이미 구현됨 (`packages/core/src/markdown.ts:29-31`)

### FR-5: WordPress 발행

**FR-5.1**: 한글 포스트 발행
- 기존 워크플로우 그대로 진행
- WordPress REST API를 통해 포스트 생성

**FR-5.2**: 영어 포스트 발행
- 번역된 콘텐츠로 새 포스트 생성
- 슬러그에 `-en` 접미사 자동 추가
- 동일한 상태(`status`)로 발행 (draft/publish)

**FR-5.3**: Polylang 언어 연결
- Polylang REST API Helper 플러그인 사용
- 한글 포스트 ID와 영어 포스트 ID 연결
- `POST /wp-json/polylang-helper/v1/link-translations`

### FR-6: 에러 처리

**FR-6.1**: 번역 API 실패
- Claude API 호출 실패 시 한글 포스트만 발행
- 에러 메시지: "번역 API 호출 실패. 한글 포스트만 발행되었습니다."
- 실패 원인을 로그에 기록

**FR-6.2**: 번역 품질 검증 실패
- 품질 검증 실패 시 한글 포스트만 발행
- 에러 메시지: "번역 품질 검증 실패. 한글 포스트만 발행되었습니다. [실패 이유]"
- 검증 실패 세부 내용 로그 기록

**FR-6.3**: WordPress 발행 실패
- 영어 포스트 발행 실패 시에도 한글 포스트는 유지
- 에러 메시지: "영어 포스트 발행 실패. 한글 포스트는 정상 발행되었습니다."

**FR-6.4**: Polylang 연결 실패
- 언어 연결 실패 시에도 두 포스트는 유지
- 경고 메시지: "Polylang 언어 연결 실패. 수동으로 연결이 필요합니다."
- 한글 포스트 ID, 영어 포스트 ID 출력

## 5. 비목표 (범위 외)

다음 기능들은 v1에서 명시적으로 **제외**됩니다:

1. **다언어 지원 확장**: 일본어, 중국어 등 추가 언어는 v1에서 제외 (향후 개선 예정)
2. **비동기 번역 처리**: 백그라운드 번역은 v1에서 제외 (현재는 동기 처리)
3. **번역 캐싱**: 동일 콘텐츠 재번역 방지 기능은 v1에서 제외
4. **번역 커스터마이징**: 전문 용어 사전 기능은 v1에서 제외
5. **번역 품질 대시보드**: 메트릭 시각화는 v1에서 제외
6. **A/B 테스팅**: 번역 프롬프트 A/B 테스팅은 v1에서 제외
7. **수동 번역 편집**: 번역 후 수동 편집 워크플로우는 v1에서 제외

```bash
blog publish content/posts/ko/my-post.md
```

**출력 예시**:
```
✓ 마크다운 파싱 완료
✓ 한글 포스트 발행 완료 (ID: 123)
  URL: https://beomanro.com/ko/my-post/
↻ 영어 번역 생성 중...
✓ 번역 완료 (2분 14초)

  - 라인 수: 520 → 541 (104%)
  - SEO 키워드: 12/12 유지
  - 제목 길이: 58자
✓ 영어 포스트 발행 완료 (ID: 124)
  URL: https://beomanro.com/en/my-post-en/
✓ Polylang 언어 연결 완료
```

```bash
blog publish content/posts/ko/my-post.md --no-translate

```
```
✓ 마크다운 파싱 완료
```

## 7. 기술 고려사항

### 아키텍처

```
blog publish content/posts/ko/my-post.md
↓
CLI (packages/cli/src/commands/publish.ts)
↓
┌─ parseMarkdownFile() [packages/core/src/markdown.ts]
│  └─ Extract metadata + content
├─ translatePost() [packages/core/src/translator.ts] ← TODO 구현
│  ├─ executeClaude() [packages/core/src/claude.ts] ← 이미 구현됨
│  │  └─ spawn('/opt/homebrew/bin/claude', ['-p', ...])
│  └─ validateTranslation() [packages/core/src/validation.ts] ← 신규
├─ createPost() [packages/core/src/wordpress.ts]
│  ├─ 한글 포스트 발행
│  └─ 영어 포스트 발행
└─ linkTranslations() [Polylang REST API Helper]
```

모든 구현된 클래스와 함수는 프로그래밍 언어의 네이티브 테스팅 프레임워크(Jest)를 사용한 유닛 테스트를 포함해야 합니다. 각 유닛 테스트 스위트는 다음을 다루는 최소 3개의 테스트 케이스를 포함해야 합니다:

- **정상 경로**: 가장 일반적이고 예상되는 시나리오가 의도한 대로 올바르게 작동하는지 확인합니다.

- **예외 케이스**: 잘못된 입력, 오류 조건 및 예외적인 상황의 적절한 처리를 검증합니다.
- **부작용**: 테스트 독립성(테스트가 서로 영향을 주지 않음)을 보장하고 코드가 글로벌 상태나 외부 시스템에 예상치 못한 영향을 주지 않는지 확인합니다.
