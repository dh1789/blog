## Overview

이 가이드라인은 블로그 포스트 생성 시 일관된 품질과 스타일을 유지하기 위한 규칙입니다.
포스트 #1 (92점), #2 (96점), #3 (98점), 테스트 (99점), #4-#6 (SEO 개선 필요)를 분석했습니다.

**달성 성과**:
- ✅ 99/100점 달성 (콘텐츠 품질)
- ✅ 수정 항목 35% 감소 (평균 4.67개 → 3개 → 0개)
- ✅ 링크/통계 품질 완벽 개선
- 🎯 SEO 최적화 강화 필요 (v1.2 목표)

**목표**:
- 검증된 정보만 사용
- 실행 가능한 코드 제공
- E-E-A-T 준수 (경험, 전문성, 권위성, 신뢰성)
- **SEO 최적화 강화** (키워드 밀도, 내부 링크)
- 사용자 유입 증가 → 수익 향상

## 1. Statistics and Claims (통계 및 주장)

### ✅ DO (지켜야 할 것)

- **공식 출처가 있는 통계만 사용**
  - 예: "Google 공식 문서에 따르면...", "Anthropic 연구에서..."
  - 출처 링크 포함 권장

- **맥락을 명시한 수치**
  - 예: "본 프로젝트 기준 약 80%", "실제 테스트 결과 5초 → 1초"
  - "약", "대략", "평균"같은 표현으로 정확도 조절

- **일반적 표현 사용**
  - 예: "많은 개발자들이", "상당한 성능 향상", "효과적인 방법"

### ❌ DON'T (피해야 할 것)

- **검증되지 않은 구체적 수치**
  - 예: "65% of marketers" (출처 없음)
  - 예: "개발자의 60%가" (근거 없음)

- **과장된 성과 수치**
  - AdSense CTR: 현실적 범위는 0.1-1% (과장 금지: 2-5%)
  - 성능 향상: 실제 측정 결과 기반으로만 작성

- **추측성 통계**
  - "아마도 80%는...", "대부분의 사람들이..." (근거 필요)

## 2. Code Quality (코드 품질)

### ✅ DO

- **모든 코드 실행 가능해야 함**
  - TypeScript strict 모드 준수
  - 타입 정의 완전
  - 에러 처리 포함

- **import 문 완전히 포함**
  ```typescript
  import { useState } from 'react';
  import WPAPI from 'wpapi';
  import { JSDOM } from 'jsdom';
  ```

- **package.json에 모든 의존성 명시**
  - dependencies와 devDependencies 구분
  - 버전 명시 (예: `"react": "^18.2.0"`)
  - **중요**: 코드에서 사용하는 모든 라이브러리 포함 (jsdom 누락 사례 주의)

- **예제 라이브러리 명확히 처리**
  - 코드 예제에서 사용하는 경우: package.json에 반드시 포함 + 설치 명령어 명시
  - 올바른 예제:
    ```markdown
    먼저 Zod를 설치하세요:
    ```bash
    npm install zod
    ```

    사용 예제:
    ```typescript
    import { z } from 'zod';
    const schema = z.string();
    ```
    ```
  - 설명만 하는 경우: "별도 설치 필요: npm install <library>" 명시

- **코드 블록에 언어 명시**
  ````markdown
  ```typescript
  const example: string = "Hello";
  ```
  ````

### ❌ DON'T

- **의존성 누락**
  - 예: jsdom 사용하면서 package.json에 없음 (포스트 #3 실수)
  - 예: zod, ora, inquirer 예제 코드 작성하면서 package.json에 누락 (테스트 포스트 실수)
  - 해결: 코드 예제 사용 시 package.json에 포함 + 설치 명령어 명시

- **플레이스홀더 코드**
  - 예: `// TODO: 구현 필요`, `// ...more code`

- **미완성 예제**
  - 예: import 문 없이 코드만, 타입 정의 없이 사용

- **하드코딩된 값**
  - 예: API 키, 비밀번호 (환경 변수 사용 권장)

## 3. Links and References (링크 및 참조)

### ✅ DO

- **공식 문서 링크 우선**
  - Anthropic Claude API: https://docs.anthropic.com
  - WordPress REST API: https://developer.wordpress.org/rest-api
  - TypeScript: https://www.typescriptlang.org

- **시리즈 포스트 명확히 참조**
  - 예: "이전 포스트 '제목'에서 다룬 내용..."
  - 발행 전이라면: "시리즈 이전 포스트 참조"

- **실제 존재하는 URL만 사용**
  - 링크 추가 전 반드시 확인

### ❌ DON'T

- **더미 링크**
  - 예: `https://github.com/example/blog-automation` (존재하지 않음)
  - 예: `https://example.com`

- **플레이스홀더**
  - 예: `[제목](내부-링크)`, `[제목](URL-to-be-added)`

- **깨진 링크**
  - 404 에러 나는 링크 금지

## 4. SEO Best Practices

### ✅ DO

- **Frontmatter 완전히 작성**
  ```yaml
  ---
  title: "명확하고 키워드 포함된 제목"
  slug: "seo-friendly-url-slug"
  excerpt: "150자 이내 요약"
  categories: ["카테고리1", "카테고리2"]
  tags: ["태그1", "태그2", "태그3"]
  status: "publish"
  language: "ko"
  date: "2025-10-28"
  ---
  ```

- **slug 명시적 설정**
  - 영문 소문자, 하이픈으로 단어 구분
  - 예: `"ai-blog-automation-guide"`

- **excerpt 최적화**
  - 150자 이내 (Google 검색 결과 표시 범위)
  - 핵심 키워드 포함
  - 명확하고 매력적인 요약

- **카테고리 2-3개, 태그 3-5개**
  - 너무 많으면 희석, 너무 적으면 분류 부족

- **키워드 밀도 최적화 (NEW in v1.2)**
  - **목표 범위**: 각 주요 키워드당 0.5-2.5%
  - **측정 방법**: (키워드 출현 횟수 / 전체 단어 수) × 100
  - **자연스러운 배치**:
    - 제목 (H1): 반드시 포함
    - 소제목 (H2/H3): 2-3개 소제목에 포함
    - 본문: 자연스럽게 5-10회 반복
    - 첫 단락: 핵심 키워드 1회 이상
    - 마지막 단락: 핵심 키워드 1회 이상
  - **변형 사용**: 동의어, 관련어를 활용하여 자연스럽게

- **내부 링크 전략 (NEW in v1.2)**
  - **시리즈 연결**: 이전/다음 포스트 명확히 링크
    - 예: "이전 포스트 [제목](실제-URL)에서..."
  - **관련 주제 링크**: 2-3개 관련 포스트 링크
  - **앵커 텍스트**: 키워드 포함하여 자연스럽게
  - **링크 위치**: 본문 내 문맥에 맞게 배치

- **메타 설명 강화 (NEW in v1.2)**
  - excerpt에 핵심 키워드 2-3개 자연스럽게 포함
  - 실행 가능한 가치 제안 (예: "~하는 방법", "~완벽 가이드")
  - 클릭 유도 문구 (예: "자세히 알아보세요", "지금 확인하세요")

### ❌ DON'T

- **키워드 스터핑**
  - 예: "AI 블로그 자동화 AI 자동화 블로그 AI..." (부자연스러움)
  - 키워드 밀도 >3%는 과도함

- **키워드 밀도 부족 (NEW in v1.2)**
  - 예: 키워드 밀도 <0.5%는 SEO 효과 미미
  - 포스트 #4-#6 문제: 대부분 0-0.3% (목표: 0.5-2.5%)

- **내부 링크 누락 (NEW in v1.2)**
  - 시리즈 포스트인데 이전 포스트 링크 없음
  - 관련 주제 포스트가 있는데 연결 안 함

- **slug 누락**
  - 자동 생성에 의존하면 최적화 어려움

- **excerpt 너무 길거나 짧음**
  - 너무 길면: 잘림 (200자+)
  - 너무 짧으면: 정보 부족 (50자-)

## 5. E-E-A-T Compliance (경험, 전문성, 권위성, 신뢰성)

### ✅ DO

- **AI 사용 투명하게 고지**
  ```markdown
  > 💡 이 글은 AI 도구(Claude)의 도움을 받아 작성되었으며, 모든 코드는 테스트를 거쳤습니다.
  ```

- **실제 경험 기반 내용**
  - 예: "직접 구현하여 1개월간 운영한 결과..."
  - 예: "실제 프로젝트에서 발생한 문제와 해결 방법..."

- **윤리적 가이드 포함**
  - AI 콘텐츠 사용 시 투명성
  - Google 정책 준수 명시
  - 저작권 존중

- **전문성 입증**
  - 기술적 깊이
  - 실용적 예제
  - 베스트 프랙티스

### ❌ DON'T

- **AI 사용 숨기기**
  - Google은 AI 콘텐츠 자체를 문제 삼지 않지만, 투명성 요구

- **검증 없는 주장**
  - 테스트하지 않은 코드
  - 경험 없는 조언

- **표절**
  - 다른 블로그/문서 복사 금지

## 6. Structure and Readability (구조 및 가독성)

### ✅ DO

- **명확한 H2/H3 계층 구조**
  ```markdown
  ## 1. 주요 섹션
  ### 1.1 하위 섹션
  ### 1.2 하위 섹션
  ## 2. 다음 주요 섹션
  ```

- **코드 블록에 언어 명시**
  - TypeScript, JavaScript, Bash, JSON 등

- **실용적 예제**
  - 독자가 바로 복사-붙여넣기 가능
  - 완전한 컨텍스트 포함

- **주의사항/팁 박스 활용**
  ```markdown
  > ⚠️ 주의: ...
  > 💡 팁: ...
  > ℹ️ 참고: ...
  ```

- **적절한 단락 구분**
  - 한 단락: 1-3 문장
  - 빈 줄로 단락 구분

### ❌ DON'T

- **너무 긴 단락**
  - 5문장 이상은 분리 고려

- **제목 없는 긴 섹션**
  - 300단어 이상은 하위 제목 추가

- **코드 설명 부족**
  - 주석이나 설명 없는 복잡한 코드

## 7. User Preferences (사용자 피드백 기반)

포스트 #1-#3을 통해 학습한 사용자 선호도:

### ✅ 항상 적용

- **품질 최우선**: Ultrathink 팩트체크 수준의 정확도
- **완벽주의**: 권장 수정까지 모두 적용 (옵션 B 선호)
- **실용성**: 통합형, 실전 적용 가능한 주제
- **빠른 진행**: 검토 후 즉시 발행 가능한 완성도

### ✅ 콘텐츠 스타일

- **깊이 있는 기술 블로그**
  - 표면적 설명보다 구현 세부사항
  - "왜"와 "어떻게"를 모두 설명

- **완전한 예제**
  - 부분 코드가 아닌 전체 컨텍스트
  - 의존성 명시
  - 에러 처리 포함

- **시리즈 연계**
  - 각 포스트는 독립적이면서도
  - 이전 포스트와 자연스러운 연결

## 8. Common Mistakes to Avoid (자주 발생하는 실수)

### 포스트 #1에서 배운 것
- ❌ GitHub 플레이스홀더 링크 (`github.com/example`)
- ❌ 검증되지 않은 통계
- ✅ 수정 후: 92/100 점

### 포스트 #2에서 배운 것
- ❌ 내부 링크 플레이스홀더
- ❌ 일부 통계 미검증
- ✅ 수정 후: 96/100 점

### 포스트 #3에서 배운 것
- ❌ AdSense CTR 과대평가 (2-5% → 0.5-1.5%)
- ❌ jsdom 의존성 누락
- ❌ 환경 변수 설명 불완전
- ✅ 수정 후: 98/100 점

### 테스트 포스트 (가이드라인 v1.1 적용)
- ✅ **초기 점수: 98/100** (수정 전)
- ❌ zod, ora, inquirer 의존성 누락 (3개)
- ❌ 설치 명령어 누락
- ✅ **최종 점수: 99/100** (수정 후)
- 📊 **가이드라인 효과**: 수정 항목 35% 감소 (평균 4.67개 → 3개 → 0개)

### 포스트 #4-#6 (가이드라인 v1.1 적용 후 발견된 SEO 문제)
- ✅ **콘텐츠 품질**: 코드, 링크, 통계 모두 검증 완료
- ❌ **SEO 점수 낮음**: #4 (78), #5 (61), #6 (40)
- 🔍 **주요 문제**:
  - 키워드 밀도 부족 (0-0.3% vs 목표 0.5-2.5%)
  - 내부 링크 전략 부재
  - 메타 설명 최적화 부족
- 📊 **개선 필요**: v1.2에서 SEO 규칙 강화

### 달성된 목표 및 다음 목표
- ✅ **99/100 점 달성**: 콘텐츠 품질 (v1.1)
- ✅ **수정 항목 최소화**: 3개 → 0개로 감소
- 🎯 **v1.2 목표**: SEO 점수 80+ 달성 (키워드 밀도 0.5-2.5%)
- 🎯 **최종 목표**: 첫 시도에 콘텐츠 99/100 + SEO 80+

## 10. Long Posts SEO Strategy (긴 포스트 SEO 전략, NEW in v1.3)

긴 포스트(1000+ 줄)는 키워드 밀도 달성이 어렵습니다. 길이에 따라 목표를 조정합니다.

### 포스트 길이별 키워드 밀도 공식

**기본 원칙**: 포스트가 길어질수록 목표 키워드 밀도를 낮춥니다.

```typescript
function calculateTargetDensity(
  baselineDensity: number,  // 기본 밀도 (0.5-2.5%)
  postLength: number        // 줄 수
): number {
  if (postLength < 500) {
    // 짧은 포스트: 기본 밀도 유지
    return baselineDensity;
  } else if (postLength < 1000) {
    // 중간 포스트: 10% 감소
    return baselineDensity * 0.9;
  } else if (postLength < 1500) {
    // 긴 포스트: 20% 감소
    return baselineDensity * 0.8;
  } else {
    // 매우 긴 포스트: 30% 감소
    return baselineDensity * 0.7;
  }
}
```

**예시**:
- 짧은 포스트 (400줄): 목표 밀도 1.0% → 1.0% (변화 없음)
- 중간 포스트 (700줄): 목표 밀도 1.0% → 0.9%
- 긴 포스트 (1200줄): 목표 밀도 1.0% → 0.8%
- 매우 긴 포스트 (1600줄): 목표 밀도 1.0% → 0.7%

### 긴 포스트 키워드 배치 전략

**섹션별 최소 키워드 출현 횟수**:

- **제목 (H1)**: 1회 필수
  - 예: "Node.js CLI 도구 개발 입문"

- **소제목 (H2/H3)**: 각 섹션당 1-2회
  - H2가 10개라면, 최소 10회 ~ 최대 20회
  - 자연스럽게 분산

- **첫 단락**: 1-2회
  - 첫 문장 또는 두 번째 문장에 포함
  - 포스트 주제를 명확히 제시

- **마지막 단락**: 1-2회
  - 결론에서 핵심 키워드 재강조
  - "이 글에서는 Node.js CLI 개발을..."

- **본문**: 섹션당 2-3회
  - 전체 본문에 균등 분산
  - 과도한 집중 방지

**1500줄 포스트 예시**:
- H1: 1회
- H2 10개 × 1.5회 = 15회
- 첫 단락: 2회
- 마지막 단락: 2회
- 본문 10개 섹션 × 2.5회 = 25회
- **총합**: 약 45회 (적절한 밀도)

### 섹션별 키워드 분포 가이드

**균등 분포 원칙**:
- 모든 주요 섹션에 키워드 포함
- 한 섹션에 5회 이상 반복 지양
- 자연스러운 문맥에서 사용

**나쁜 예시** (키워드 집중):
```markdown
## Node.js 설치
Node.js를 설치합니다. Node.js는 Node.js 공식 사이트에서
다운로드할 수 있습니다. Node.js 설치 후 Node.js를 확인하세요.
(5회 반복 - 부자연스러움)
```

**좋은 예시** (자연스러운 배치):
```markdown
## Node.js 설치
Node.js를 설치합니다. 공식 사이트에서 다운로드할 수 있으며,
설치 후 버전을 확인하세요. 이 도구를 사용하면...
(1회 - 자연스러움, 다른 섹션에 분산)
```

**자연스러운 변형 활용**:
- 원형: "Node.js CLI"
- 변형: "CLI 도구", "명령줄 도구", "Node.js 도구"
- LSI 키워드: "TypeScript", "Commander.js", "npm"

### SEO 점수 목표 설정 (길이별)

포스트 길이에 따라 현실적인 SEO 목표를 설정합니다.

- **짧은 포스트 (< 500줄)**: 목표 SEO 80+
  - 키워드 밀도 달성 용이
  - 최고 점수 가능

- **중간 포스트 (500-1000줄)**: 목표 SEO 75+
  - 적절한 밀도 유지 가능
  - 고품질 달성 가능

- **긴 포스트 (1000-1500줄)**: 목표 SEO 70+
  - 키워드 밀도 달성 어려움
  - 콘텐츠 품질로 보완

- **매우 긴 포스트 (1500+ 줄)**: 목표 SEO 65+
  - 키워드 밀도보다 품질 우선
  - 섹션 분포가 더 중요

**이유**:
- 긴 포스트는 키워드 밀도 달성이 물리적으로 어려움
- 과도한 키워드 반복은 오히려 역효과
- 콘텐츠 품질과 사용자 경험이 우선
- 내부 링크와 섹션 구조로 SEO 보완

### 긴 포스트 최적화 체크리스트

- [ ] 포스트 길이 확인 (줄 수)
- [ ] 길이별 목표 키워드 밀도 계산
- [ ] H2 제목에 키워드 분산 (2-3개 이상)
- [ ] 첫 단락에 핵심 키워드 포함
- [ ] 마지막 단락에 핵심 키워드 재강조
- [ ] 한 섹션에 키워드 5회 이상 반복 없는지 확인
- [ ] 자연스러운 변형 활용 (동의어, LSI 키워드)
- [ ] 길이에 맞는 현실적 SEO 목표 설정
- [ ] 내부 링크로 SEO 보완 (3-5개)
- [ ] 섹션 구조 명확 (목차, H2/H3 계층)
