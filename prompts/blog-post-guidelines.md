# Blog Post Generation Guidelines

**Version**: v1.1 (2025-10-28)
**Based on**: 포스트 #1-#3 분석 + 테스트 포스트 검증 및 Ultrathink 팩트체크 결과

## Overview

이 가이드라인은 블로그 포스트 생성 시 일관된 품질과 스타일을 유지하기 위한 규칙입니다.
포스트 #1 (92점), #2 (96점), #3 (98점)을 분석하고, 테스트 포스트 (99점)로 효과를 검증했습니다.

**달성 성과**:
- ✅ 99/100점 달성 (테스트 포스트)
- ✅ 수정 항목 35% 감소 (평균 4.67개 → 3개 → 0개)
- ✅ 링크/통계 품질 완벽 개선

**목표**:
- 검증된 정보만 사용
- 실행 가능한 코드 제공
- E-E-A-T 준수 (경험, 전문성, 권위성, 신뢰성)
- SEO 최적화
- 사용자 유입 증가 → 수익 향상

## 1. Statistics and Claims (통계 및 주장)

### ✅ DO (지켜야 할 것)

- **공식 출처가 있는 통계만 사용**
  - 예: "Google 공식 문서에 따르면...", "Anthropic 연구에서..."
  - 출처 링크 포함 권장

- **맥락을 명시한 수치**
  - 예: "본 프로젝트 기준 약 80%", "실제 테스트 결과 5초 → 1초"
  - "약", "대략", "평균"같은 표현으로 정확도 조절

- **일반적 표현 사용**
  - 예: "많은 개발자들이", "상당한 성능 향상", "효과적인 방법"

### ❌ DON'T (피해야 할 것)

- **검증되지 않은 구체적 수치**
  - 예: "65% of marketers" (출처 없음)
  - 예: "개발자의 60%가" (근거 없음)

- **과장된 성과 수치**
  - AdSense CTR: 현실적 범위는 0.1-1% (과장 금지: 2-5%)
  - 성능 향상: 실제 측정 결과 기반으로만 작성

- **추측성 통계**
  - "아마도 80%는...", "대부분의 사람들이..." (근거 필요)

## 2. Code Quality (코드 품질)

### ✅ DO

- **모든 코드 실행 가능해야 함**
  - TypeScript strict 모드 준수
  - 타입 정의 완전
  - 에러 처리 포함

- **import 문 완전히 포함**
  ```typescript
  import { useState } from 'react';
  import WPAPI from 'wpapi';
  import { JSDOM } from 'jsdom';
  ```

- **package.json에 모든 의존성 명시**
  - dependencies와 devDependencies 구분
  - 버전 명시 (예: `"react": "^18.2.0"`)
  - **중요**: 코드에서 사용하는 모든 라이브러리 포함 (jsdom 누락 사례 주의)

- **예제 라이브러리 명확히 처리**
  - 코드 예제에서 사용하는 경우: package.json에 반드시 포함 + 설치 명령어 명시
  - 올바른 예제:
    ```markdown
    먼저 Zod를 설치하세요:
    ```bash
    npm install zod
    ```

    사용 예제:
    ```typescript
    import { z } from 'zod';
    const schema = z.string();
    ```
    ```
  - 설명만 하는 경우: "별도 설치 필요: npm install <library>" 명시

- **코드 블록에 언어 명시**
  ````markdown
  ```typescript
  const example: string = "Hello";
  ```
  ````

### ❌ DON'T

- **의존성 누락**
  - 예: jsdom 사용하면서 package.json에 없음 (포스트 #3 실수)
  - 예: zod, ora, inquirer 예제 코드 작성하면서 package.json에 누락 (테스트 포스트 실수)
  - 해결: 코드 예제 사용 시 package.json에 포함 + 설치 명령어 명시

- **플레이스홀더 코드**
  - 예: `// TODO: 구현 필요`, `// ...more code`

- **미완성 예제**
  - 예: import 문 없이 코드만, 타입 정의 없이 사용

- **하드코딩된 값**
  - 예: API 키, 비밀번호 (환경 변수 사용 권장)

## 3. Links and References (링크 및 참조)

### ✅ DO

- **공식 문서 링크 우선**
  - Anthropic Claude API: https://docs.anthropic.com
  - WordPress REST API: https://developer.wordpress.org/rest-api
  - TypeScript: https://www.typescriptlang.org

- **시리즈 포스트 명확히 참조**
  - 예: "이전 포스트 '제목'에서 다룬 내용..."
  - 발행 전이라면: "시리즈 이전 포스트 참조"

- **실제 존재하는 URL만 사용**
  - 링크 추가 전 반드시 확인

### ❌ DON'T

- **더미 링크**
  - 예: `https://github.com/example/blog-automation` (존재하지 않음)
  - 예: `https://example.com`

- **플레이스홀더**
  - 예: `[제목](내부-링크)`, `[제목](URL-to-be-added)`

- **깨진 링크**
  - 404 에러 나는 링크 금지

## 4. SEO Best Practices

### ✅ DO

- **Frontmatter 완전히 작성**
  ```yaml
  ---
  title: "명확하고 키워드 포함된 제목"
  slug: "seo-friendly-url-slug"
  excerpt: "150자 이내 요약"
  categories: ["카테고리1", "카테고리2"]
  tags: ["태그1", "태그2", "태그3"]
  status: "publish"
  language: "ko"
  date: "2025-10-28"
  ---
  ```

- **slug 명시적 설정**
  - 영문 소문자, 하이픈으로 단어 구분
  - 예: `"ai-blog-automation-guide"`

- **excerpt 최적화**
  - 150자 이내 (Google 검색 결과 표시 범위)
  - 핵심 키워드 포함
  - 명확하고 매력적인 요약

- **카테고리 2-3개, 태그 3-5개**
  - 너무 많으면 희석, 너무 적으면 분류 부족

- **자연스러운 키워드 사용**
  - 키워드 스터핑 피하기
  - 문맥에 맞게 자연스럽게 사용

### ❌ DON'T

- **키워드 스터핑**
  - 예: "AI 블로그 자동화 AI 자동화 블로그 AI..." (부자연스러움)

- **slug 누락**
  - 자동 생성에 의존하면 최적화 어려움

- **excerpt 너무 길거나 짧음**
  - 너무 길면: 잘림 (200자+)
  - 너무 짧으면: 정보 부족 (50자-)

## 5. E-E-A-T Compliance (경험, 전문성, 권위성, 신뢰성)

### ✅ DO

- **AI 사용 투명하게 고지**
  ```markdown
  > 💡 이 글은 AI 도구(Claude)의 도움을 받아 작성되었으며, 모든 코드는 테스트를 거쳤습니다.
  ```

- **실제 경험 기반 내용**
  - 예: "직접 구현하여 1개월간 운영한 결과..."
  - 예: "실제 프로젝트에서 발생한 문제와 해결 방법..."

- **윤리적 가이드 포함**
  - AI 콘텐츠 사용 시 투명성
  - Google 정책 준수 명시
  - 저작권 존중

- **전문성 입증**
  - 기술적 깊이
  - 실용적 예제
  - 베스트 프랙티스

### ❌ DON'T

- **AI 사용 숨기기**
  - Google은 AI 콘텐츠 자체를 문제 삼지 않지만, 투명성 요구

- **검증 없는 주장**
  - 테스트하지 않은 코드
  - 경험 없는 조언

- **표절**
  - 다른 블로그/문서 복사 금지

## 6. Structure and Readability (구조 및 가독성)

### ✅ DO

- **명확한 H2/H3 계층 구조**
  ```markdown
  ## 1. 주요 섹션
  ### 1.1 하위 섹션
  ### 1.2 하위 섹션
  ## 2. 다음 주요 섹션
  ```

- **코드 블록에 언어 명시**
  - TypeScript, JavaScript, Bash, JSON 등

- **실용적 예제**
  - 독자가 바로 복사-붙여넣기 가능
  - 완전한 컨텍스트 포함

- **주의사항/팁 박스 활용**
  ```markdown
  > ⚠️ 주의: ...
  > 💡 팁: ...
  > ℹ️ 참고: ...
  ```

- **적절한 단락 구분**
  - 한 단락: 1-3 문장
  - 빈 줄로 단락 구분

### ❌ DON'T

- **너무 긴 단락**
  - 5문장 이상은 분리 고려

- **제목 없는 긴 섹션**
  - 300단어 이상은 하위 제목 추가

- **코드 설명 부족**
  - 주석이나 설명 없는 복잡한 코드

## 7. User Preferences (사용자 피드백 기반)

포스트 #1-#3을 통해 학습한 사용자 선호도:

### ✅ 항상 적용

- **품질 최우선**: Ultrathink 팩트체크 수준의 정확도
- **완벽주의**: 권장 수정까지 모두 적용 (옵션 B 선호)
- **실용성**: 통합형, 실전 적용 가능한 주제
- **빠른 진행**: 검토 후 즉시 발행 가능한 완성도

### ✅ 콘텐츠 스타일

- **깊이 있는 기술 블로그**
  - 표면적 설명보다 구현 세부사항
  - "왜"와 "어떻게"를 모두 설명

- **완전한 예제**
  - 부분 코드가 아닌 전체 컨텍스트
  - 의존성 명시
  - 에러 처리 포함

- **시리즈 연계**
  - 각 포스트는 독립적이면서도
  - 이전 포스트와 자연스러운 연결

## 8. Common Mistakes to Avoid (자주 발생하는 실수)

### 포스트 #1에서 배운 것
- ❌ GitHub 플레이스홀더 링크 (`github.com/example`)
- ❌ 검증되지 않은 통계
- ✅ 수정 후: 92/100 점

### 포스트 #2에서 배운 것
- ❌ 내부 링크 플레이스홀더
- ❌ 일부 통계 미검증
- ✅ 수정 후: 96/100 점

### 포스트 #3에서 배운 것
- ❌ AdSense CTR 과대평가 (2-5% → 0.5-1.5%)
- ❌ jsdom 의존성 누락
- ❌ 환경 변수 설명 불완전
- ✅ 수정 후: 98/100 점

### 테스트 포스트 (가이드라인 적용 후)
- ✅ **초기 점수: 98/100** (수정 전)
- ❌ zod, ora, inquirer 의존성 누락 (3개)
- ❌ 설치 명령어 누락
- ✅ **최종 점수: 99/100** (수정 후)
- 📊 **가이드라인 효과**: 수정 항목 35% 감소 (평균 4.67개 → 3개 → 0개)

### 달성된 목표
- ✅ **99/100 점 달성**: 가이드라인 시스템으로 달성
- ✅ **수정 항목 최소화**: 3개 → 0개로 감소
- 🎯 **다음 목표**: 첫 시도에 99/100점 (수정 0-1개)

## 9. Quality Checklist (품질 체크리스트)

포스트 작성 후 반드시 확인:

### 통계 및 수치
- [ ] 모든 통계에 출처 있음 또는 일반적 표현 사용
- [ ] 과장되지 않은 현실적 수치
- [ ] AdSense CTR 등 업계 표준 준수

### 코드
- [ ] 모든 코드 실행 가능
- [ ] import 문 완전
- [ ] package.json에 의존성 모두 명시
- [ ] 예제 라이브러리 설치 명령어 명시
- [ ] 타입 정의 완전

### 링크
- [ ] 플레이스홀더 없음
- [ ] 모든 링크 실제 존재 확인
- [ ] 공식 문서 우선

### SEO
- [ ] title, slug, excerpt, categories, tags 완전
- [ ] slug 명시적 설정
- [ ] excerpt 150자 이내

### E-E-A-T
- [ ] AI 사용 고지
- [ ] 실제 경험 기반
- [ ] 윤리적 가이드 포함

### 구조
- [ ] H2/H3 계층 명확
- [ ] 코드 블록 언어 명시
- [ ] 적절한 단락 구분
