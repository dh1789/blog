## 1. 개요

블로그 포스트의 시각적 품질을 향상시키고 사용자 참여도를 높이기 위해, AI를 활용한 이미지 자동 생성 기능을 개발합니다. 이 기능은 블로거가 별도의 이미지 편집 작업 없이도 전문적인 품질의 이미지를 포스트에 삽입할 수 있도록 지원하여, 콘텐츠 제작 시간을 단축하고 블로그의 전반적인 수익성을 향상시키는 것을 목표로 합니다.

**핵심 가치**:
- ⏱️ 콘텐츠 제작 시간 50% 단축 (이미지 수동 작업 제거)
- 📈 사용자 참여도 향상 (체류 시간 +30-50%, Bounce rate -25-35%)
- 💰 간접적 수익 개선 (참여도 증가를 통한 AdSense RPM 향상)
- 🌍 다국어 콘텐츠 일관성 유지 (한/영 동일 이미지)

**팩트 기반 근거**:
- ✅ **참여도 증가**: 이미지 있는 페이지는 34% bounce rate 감소, 최대 3분 추가 체류 ([출처](https://www.searchenginejournal.com))
- ✅ **RPM 상관관계**: "The higher the time on-site, the better the CPMs will be" ([출처](https://www.mile.tech/blog/adsense-cpm-rates-2024))
- ⚠️ **수익 직접 증거 없음**: 이미지 → AdSense 수익 직접 증가는 입증 안 됨 (간접 효과만 확인)

## 2. 목표

### 주요 목표
1. **자동화**: `blog publish --generate-images` 명령어로 이미지 생성부터 삽입까지 원클릭 자동화
2. **최적화**: 수익성 높은 블로그 벤치마크 기반으로 이미지 개수, 크기, 품질 자동 최적화
3. **통합**: Epic 12.0 WordPress 미디어 라이브러리 기능과 완벽 통합
4. **안정성**: Claude Code CLI 이미지 생성 기능 검증 및 고도화

### 성공 지표

**Primary KPI** (직접 측정):
- 평균 체류 시간 (Time on Page): +30-50% 목표
- Bounce Rate: -25-35% 목표
- 페이지뷰: +20-30% 목표

**Secondary KPI** (기술적 성능):
- 이미지 생성 성공률 ≥90%
- 평균 생성 시간 ≤30초/이미지
- 포스트 발행 시간 증가 ≤20%

**Lagging KPI** (간접 효과, 3개월 후 측정):
- AdSense RPM 변화율 (참여도 증가에 따른 간접 효과)

## 4. 기능 요구사항

### FR-13.1: CLI 통합
1. `blog publish` 명령어에 `--generate-images` 플래그 추가
2. 플래그 활성화 시 자동으로 이미지 생성 프로세스 실행
3. 기존 publish 워크플로우와 매끄럽게 통합

### FR-13.2: 벤치마크 기반 최적화 (1회성 조사)

**목적**: 수익성 높은 블로그의 이미지 전략을 분석하여 최적 설정값을 도출하고, 이를 config 파일에 저장하여 모든 포스트에 일관되게 적용

**실행 시점**: 개발 단계 또는 초기 설정 시 **1회만** 실행 (비용 및 시간 절약)

**벤치마크 프로세스** (개발 단계):

1. **Web 검색 및 데이터 수집**:
   - SEO 상위 블로그 (tech, 개발, WordPress 관련) 크롤링
   - 포스트당 평균 이미지 개수 수집
   - 이미지 크기 (width × height) 분석
   - 파일 크기 및 포맷 분석
   - **예상 시간**: 30-60분
   - **예상 비용**: Web 크롤링 무료, AI 분석 $0.50-$1.00

2. **AI 분석**:
   - Claude를 활용하여 수집된 데이터 패턴 분석
   - 포스트 길이별 최적 이미지 개수 추출
   - 이미지 배치 위치 전략 도출 (예: H2 헤딩마다, 문단 N개마다)
   - 권장 설정값 생성

3. **설정 파일 저장**:
   - 분석 결과를 `config/image-defaults.json`에 저장
   - 포스트 길이에 따른 이미지 개수 규칙
   - 이미지 크기, 포맷, 압축률 기본값
   - 배치 위치 전략

**런타임 동작** (포스트 발행 시):
- 저장된 `config/image-defaults.json` 설정값만 사용
- Web 검색이나 AI 분석 **실행하지 않음**
- 즉시 이미지 생성 시작 (빠른 실행)

**권장 설정값 예시** (벤치마크 결과 기반):
```json
{
  "imageDefaults": {
    "featuredImage": {
      "width": 1200,
      "height": 630,
      "format": "webp",
      "quality": 85
    },
    "contentImages": {
      "countRule": "1 image per 500 words",
      "placement": "after every 2nd H2 heading",
      "width": 800,
      "height": 450,
      "format": "webp",
      "quality": 80
    }
  }
}
```

**재벤치마크** (선택 사항):
- 필요 시 `blog benchmark-images` 명령어로 수동 재실행
- 블로그 전략 변경 또는 업계 트렌드 변화 시
- 권장 주기: 6개월-1년

### FR-13.3: Claude Code CLI 이미지 생성 스파이크 (1회성 검증)

**목적**: Claude Code CLI의 이미지 생성 기능을 검증하여 최적 설정값과 제약사항을 파악하고, 이를 문서화하여 개발 시 참고 자료로 활용

**실행 시점**: 개발 단계 **1회만** 실행 (Phase 1에서 완료 후 문서화)

**검증 프로세스** (개발 단계):

1. **기본 이미지 생성 가능 여부**:
   - Claude Code CLI로 간단한 프롬프트 전달
   - 이미지 파일 생성 및 저장 확인
   - 실패 케이스 테스트 (타임아웃, API 에러)
   - **결과 문서화**: `tasks/spike-results-image-generation.md`

2. **품질/크기 제어 가능성**:
   - 해상도 지정: `--width 1200 --height 630`
   - 품질 설정: `--quality high/medium/low`
   - 파일 포맷 지정: `--format webp/png/jpg`
   - 실제 출력 파일 검증
   - **결과 저장**: 지원 가능한 옵션 리스트화

3. **비용 및 속도 측정**:
   - 이미지당 생성 시간 측정 (평균, 최대, 최소)
   - API 호출 비용 추정 (요청당 비용)
   - 동시 생성 가능 개수 테스트
   - **결과 문서화**: 예상 비용 및 성능 기준

4. **블로그 컨텍스트 기반 생성**:
   - 포스트 제목 + excerpt를 프롬프트에 포함
   - 생성된 이미지의 관련성 평가
   - 키워드 기반 스타일 지정 (예: "technical diagram", "screenshot")
   - 실패 케이스: 부적절한 콘텐츠 생성 방지
   - **결과 저장**: 최적 프롬프트 템플릿 작성

**스파이크 성공 기준**:
- 4가지 검증 항목 모두 "가능" 판정
- 평균 생성 시간 ≤30초
- 이미지당 비용 ≤$0.10
- 관련성 점수 ≥4/5

**검증 결과 저장**:
- `tasks/spike-results-image-generation.md`: 전체 검증 결과 상세 문서
- `config/claude-code-cli-limits.json`: API 제약사항 및 최적 설정값
- 프롬프트 템플릿: `FR-13.4`에 반영

**런타임 동작** (포스트 발행 시):
- 스파이크 검증 **실행하지 않음**
- 저장된 설정값과 프롬프트 템플릿만 사용
- 즉시 이미지 생성 시작 (빠른 실행)

### FR-13.4: 이미지 생성 프롬프트 고도화

**프롬프트 구조**:
```
You are creating a blog post featured image.

Context:
- Post Title: {title}
- Post Excerpt: {excerpt}
- Keywords: {tags}
- Language: {language}
- Target Audience: Tech-savvy developers and bloggers

Requirements:
- Style: Modern, clean, professional
- Color Scheme: Match WordPress Avada theme (blue/white tones)
- Text: Minimal or none (avoid language-specific text)
- Format: 1200x630px, WebP
- Quality: High resolution, optimized for web

Generate a visually appealing featured image that represents the post topic.
```

**동적 요소**:
- 포스트 카테고리별 스타일 변경 (개발/WordPress/SEO)
- 시리즈 포스트는 일관된 디자인 템플릿 사용
- 계절/트렌드 반영 (선택 사항)

### FR-13.5: 로컬 저장 및 WordPress 업로드

1. **로컬 저장**:
   - 경로: `content/posts/images/{post-slug}/`
   - 파일명 규칙: `featured.webp`, `section-1.webp`, `section-2.webp`
   - 메타데이터 저장: `images.json` (제목, 설명, 생성 시간)

2. **마크다운 삽입**:
   - Featured image: frontmatter에 `featured_image: "./images/{post-slug}/featured.webp"` 추가
   - 본문 이미지: 적절한 위치에 `![description](./images/{post-slug}/section-1.webp)` 자동 삽입
   - 이미지 캡션 자동 생성

3. **WordPress 업로드**:
   - Epic 12.0 `uploadMedia()` 함수 활용
   - 중복 이미지 감지 (`findMediaByFilename()`)
   - 업로드 성공 시 마크다운 경로를 WordPress URL로 자동 변환

### FR-13.6: 다국어 이미지 공유

1. **한글 포스트 발행 시**:
   - 이미지 생성 및 로컬 저장
   - WordPress 업로드
   - frontmatter에 이미지 경로 기록

2. **자동 번역 시**:
   - 한글 포스트의 `featured_image` 경로 복사
   - 본문 이미지 경로도 동일하게 복사
   - 영문 포스트는 새로운 이미지 생성 없이 기존 이미지 참조

3. **이미지 경로 관리**:
   - 언어별로 다른 경로가 아닌 공통 경로 사용
   - WordPress 미디어 라이브러리에서 한 번만 업로드

### FR-13.7: 에러 처리 및 로깅

1. **이미지 생성 실패**:
   - 경고 메시지 출력: "⚠️ 이미지 생성 실패: {이유}"
   - 포스트 발행 계속 진행 (이미지 없이)
   - 실패 로그 저장: `logs/image-generation-failures.log`

2. **업로드 실패**:
   - 로컬 파일은 유지 (재시도 가능)
   - 경고 메시지 출력
   - 포스트는 로컬 이미지 경로로 발행

3. **재시도 메커니즘**:
   - `blog retry-images {post-id}` 명령어 제공
   - 실패한 이미지만 선택적 재생성

## 5. 비목표 (Out of Scope)

1. ❌ **비용 제한 기능**: 이미지 생성 API 비용 제한 없음 (사용자 요청)
2. ❌ **사용자 승인 프로세스**: 미리보기 및 승인 단계 없음 (완전 자동화)
3. ❌ **이미지 편집 기능**: 생성 후 수동 편집 불가 (재생성만 가능)
4. ❌ **동영상 생성**: 정적 이미지만 지원
5. ❌ **실시간 생성**: 발행 시점에만 생성 (포스트 작성 중 미리보기 없음)

## 6. 설계 고려사항

### 아키텍처 구조
```
packages/
  core/
    src/
      # === 런타임 모듈 (매 포스트 발행 시 실행) ===
      image-generation.ts    # Claude Code CLI 래퍼
      image-optimizer.ts     # 크기/품질 최적화

      # === 개발 도구 (Phase 1에서만 1회 실행) ===
      benchmark-analyzer.ts  # Web 검색 + AI 분석 (FR-13.2)
      spike-validator.ts     # Claude Code CLI 스파이크 검증 (FR-13.3)

  cli/
    src/
      commands/
        # === 런타임 명령어 ===
        publish.ts              # --generate-images 플래그 추가

        # === 개발/관리 도구 ===
        retry-images.ts         # 재생성 명령어
        benchmark-images.ts     # 벤치마크 재실행 (선택 사항, 6개월-1년 주기)
        spike-images.ts         # 스파이크 재검증 (선택 사항)

config/
  # === 1회성 작업 결과 (git 커밋) ===
  image-defaults.json           # 벤치마크 결과 (FR-13.2)
  claude-code-cli-limits.json   # 스파이크 검증 결과 (FR-13.3)
```

**모듈 역할**:
- **런타임 모듈** (매 포스트 발행 시):
  - `image-generation.ts`: config 설정값 사용하여 이미지 생성
  - `image-optimizer.ts`: 생성된 이미지 최적화

- **개발 도구** (Phase 1에서만 1회):
  - `benchmark-analyzer.ts`: 벤치마크 조사 → `config/image-defaults.json` 생성
  - `spike-validator.ts`: 스파이크 검증 → `config/claude-code-cli-limits.json` 생성

- **관리 도구** (필요 시):
  - `benchmark-images.ts`: 6개월-1년 주기로 벤치마크 재조사

## 8. 성공 지표

### Primary KPI (직접 측정, 2주 후)
- **평균 체류 시간 (Time on Page)**: +30-50% 목표 (팩트 기반: 최대 3분 증가 가능)
- **Bounce Rate**: -25-35% 목표 (팩트 기반: 34% 감소 가능)
- **페이지뷰**: +20-30% 목표

### Secondary KPI (기술적 성능)
- **생성 성공률**: ≥90% (100회 시도 중 90회 성공)
- **평균 생성 시간**: ≤30초/이미지
- **이미지 관련성**: ≥4.0/5.0 (사용자 평가)
- **포스트 발행 시간 증가**: ≤20% (5분 → 6분 이내)

### Lagging KPI (간접 효과, 3개월 후)
- **AdSense RPM**: 변화율 측정 (참여도 증가에 따른 간접 효과)
- **AdSense CTR**: 변화율 측정
- **총 수익**: 전후 비교

### 정성적 지표
- 사용자가 수동 이미지 편집 작업을 하지 않음
- 블로그 포스트의 시각적 품질 향상
- 콘텐츠 제작 시간 단축

### 주요 리스크

1. **⚠️ 불확실한 수익 영향 (HIGH RISK)**:
   - **현상**: 이미지 → AdSense 수익 직접 상관관계 입증 안 됨
   - **근거**: Web 리서치 결과, 참여도 증가는 확인되나 수익 증가는 간접 효과만 존재
   - **완화 전략**:
     - A/B 테스트로 실제 효과 검증 (50% 포스트만 이미지 추가)
     - Primary KPI를 참여도 지표로 설정 (체류 시간, Bounce rate)
     - 3개월 후 Lagging KPI로 수익 변화 측정
     - 효과 미미 시 기능 폐기 또는 수동 이미지만 사용

2. **비용 대비 효과 불명확 (MEDIUM RISK)**:
   - **현상**: AI 이미지 생성 비용 vs 실제 수익 증가 불명확
   - **완화 전략**:
     - Phase 1 스파이크에서 정확한 비용 측정
     - 월 예산 설정 (예: $50/월)
     - ROI 추적 (비용 vs 수익 변화)

3. **측정 어려움 (MEDIUM RISK)**:
   - **현상**: 이미지 효과만 분리하여 측정 어려움 (SEO, 콘텐츠 품질 등 다른 요인 혼재)
   - **완화 전략**:
     - A/B 테스트로 통제 그룹 설정
     - 동일 카테고리 포스트 비교
     - 장기간 (3개월) 데이터 수집
